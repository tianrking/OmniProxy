# OmniProxy 总体规划（中文）

English Version: [MASTER_PLAN.md](./MASTER_PLAN.md)

这是推进 OmniProxy 到 v1.0 的统一执行规划。

## 1. 交付原则

1. 每个里程碑必须同时包含代码、测试、文档。
2. 核心链路保持简洁高性能：代理 -> 观测 -> 分析 -> 回放 -> 篡改。
3. CLI 参数与事件合同优先保持兼容。
4. 任何高风险运行时能力必须有保护策略（超时、预算、降级）。

## 2. 当前能力快照

已完成：
1. HTTPS MITM 与动态 CA 初始化。
2. HTTP/1.1 + HTTP/2 抓包与 WS 事件输出。
3. 规则引擎（header/status/body 动作，regex-lite，首条命中优先）。
4. 回放 CLI（可编辑/交互）与 diff 哈希。
5. TUI 键盘工作流（回放 + CONNECT 噪音抑制）。
6. Wasm v1 可变更 hook（请求/响应修改载荷）。
7. 基线压测工具（`omni-bench`、`omni-converge`）。
8. 发布硬化基线（checksum/provenance/signature）。

进行中/待收口：
1. 未知长度持续流量的真正流式路径收口。
2. HTTP/1.1 + HTTP/2 在更大压测矩阵下的正确性收敛。
3. Wasm ABI 兼容矩阵与资源预算指标增强。
4. v1.0 runbook 最终闭环与证据化检查。

## 3. 里程碑与迭代评估

预计剩余迭代：**6-8 个版本**（不发生大规模架构重置前提）。

### M1（1 个迭代）：流式稳定性收口

范围：
1. 未知长度 body 流式策略继续完善。
2. 慢消费者背压行为测试。
3. 大包持续场景内存包络验证。

验收：
1. 30 分钟以上持续流量无 OOM/泄漏。
2. 截断/采样策略开启时内存上界可控。

### M2（1-2 个迭代）：协议正确性收敛

范围：
1. 扩展 `omni-converge` 场景（混合方法、keep-alive、不同响应体大小）。
2. 请求-响应关联正确性可重复验证。
3. 明确失败特征和回归阈值。

验收：
1. http1/http2pref 模式下错误率达标。
2. 多轮执行后 p95/p99 漂移在阈值内。

### M3（1-2 个迭代）：Wasm v1 硬化

范围：
1. ABI 兼容性测试覆盖历史/变体载荷。
2. CPU/内存/失败预算指标完善。
3. 插件持续异常时的安全降级模式。

验收：
1. 兼容性测试全部通过。
2. 插件异常不影响核心代理主链路稳定性。

### M4（1 个迭代）：分析师体验增强

范围：
1. TUI 内可编辑回放入口。
2. 按会话/时间窗批量重访能力。
3. 更强详情快捷操作与提示。

验收：
1. 终端内可完成“抓包->过滤->编辑->回放”的闭环。

### M5（1 个迭代）：运维与文档收口

范围：
1. runbook 在 macOS/Linux/Windows 的最终验证矩阵。
2. 端到端部署检查清单 + 故障决策树。
3. 中英文文档一致性审计。

验收：
1. 新用户仅按文档可在 10 分钟内完成首次 MITM 成功测试。

### M6（1 个迭代）：v1.0 冻结与发布签收

范围：
1. 锁定 CLI/API 合同版本说明。
2. RC 版本浸泡测试。
3. 最终签名产物与 provenance 复核。

验收：
1. 发布门禁全部通过。
2. 核心路径无 P0/P1 未解决缺陷。

## 4. 单迭代完成定义（DoD）

1. 代码合入且通过 CI 矩阵。
2. 自动化测试更新并通过。
3. 中英文文档同步更新。
4. 提交信息包含变更摘要。
5. 文档提供可执行的人工验证命令。

## 5. 风险清单

1. 大吞吐场景下 body 缓冲路径回归。
2. 多路复用边界场景关联错误。
3. 插件 ABI 漂移导致隐性不兼容。
4. 中英文文档漂移。

缓解：
1. 在 converge/bench 中固化可复现负载场景。
2. 保持事件合同与 ABI 固定测试样例。
3. 强制每个功能提交同步文档一致性更新。

## 6. 跟踪节奏

1. 每个功能批次同步更新 `docs/ROADMAP.md` 与本文件。
2. 保持小步快跑、可测试、可推送节奏。
3. 在提交说明中记录里程碑证据与验收结果。
